Case Study 01 — Directory Services

Domain Controller

CategoryActive Directory
DifficultyIntermediate
OS UsedWindows Server 2008 R2

Scenario

A small enterprise needed to transition from a peer-to-peer workgroup network to a centralized domain environment. The goal was to deploy a Domain Controller to enable centralized authentication, enforce security policies, and simplify user and computer management across all workstations.

Overview

This case study documents the full process of promoting a Windows Server 2008 R2 machine to a Domain Controller using the Active Directory Domain Services role and DCPromo wizard. It covers pre-requisite checks, DNS configuration, forest and domain setup, and post-promotion verification. The resulting domain became the authentication backbone for all subsequent configurations in the environment.

Case Study Video

Implementation Steps

  1. Verified server had a static IP address and correct DNS pointing to itself
  2. Installed the Active Directory Domain Services role via Server Manager
  3. Launched DCPromo and selected 'Create a new domain in a new forest'
  4. Set the forest root domain name (e.g., corp.local) and forest functional level
  5. Confirmed DNS Server role installation during the DCPromo process
  6. Set a strong Directory Services Restore Mode (DSRM) password
  7. Completed the wizard and rebooted — server promoted to Domain Controller
  8. Verified AD DS using Active Directory Users and Computers (ADUC)
  9. Confirmed DNS zone created automatically for the domain
  10. Tested domain reachability from a client workstation using nslookup

Key Findings

  • DCPromo automatically installs and configures DNS when no existing DNS server is detected
  • Forest Functional Level determines which advanced AD features are available across all domains
  • The DSRM password is critical for disaster recovery — must be stored securely offline
  • Post-promotion, the server's own DNS must point to 127.0.0.1 or its own IP, not external DNS
  • The first DC in the forest holds all five FSMO roles by default
  • A single DC represents a critical single point of failure — a second DC should be added promptly
  • The Default Domain Policy GPO is automatically created and linked to the domain root
  • NetLogon and SYSVOL shares are created automatically and are essential for domain operation

Outcome & Lessons Learned

The domain controller was successfully deployed and all workstations were subsequently joined to the domain. Centralizing authentication immediately reduced helpdesk overhead by enabling password resets from a single location. The key lesson learned was the importance of setting a static IP and configuring DNS correctly before running DCPromo — misconfigurations at this stage cause difficult-to-diagnose replication and authentication failures later.
