Case Study 08 — Policy Management

Group Policy Configuration

CategoryActive Directory Policy
DifficultyIntermediate–Advanced
OS UsedWindows Server 2008 R2

Scenario

The organization needed to enforce consistent security settings, restrict unauthorized software usage, and deploy standardized desktop configurations across all department workstations. Group Policy Objects were designed and deployed to address these requirements across the IT, HR, and Sales OUs.

Overview

This case study documents designing a GPO strategy, creating multiple GPOs with specific purposes, linking them to the correct OUs, testing with gpresult, and troubleshooting inheritance conflicts. Policies enforced include password complexity requirements, USB storage restrictions, desktop wallpaper, software restrictions, and automatic drive mappings via Group Policy Preferences.

Case Study Video

Implementation Steps

  1. Opened Group Policy Management Console (GPMC) from Administrative Tools
  2. Reviewed the Default Domain Policy — modified password policy: minimum 8 chars, complexity enabled, 90-day expiry
  3. Created GPO 'Security-Baseline' linked to the domain root: enabled audit policies, disabled autorun
  4. Created GPO 'USB-Restriction' linked to HR and Sales OUs: blocked removable storage via Device Installation Restrictions
  5. Created GPO 'Desktop-Standard' linked to all department OUs: set corporate wallpaper, removed Run from Start Menu
  6. Created GPO 'Drive-Mappings' using Group Policy Preferences > Drive Maps: mapped department shares based on group membership using Item-Level Targeting
  7. Created GPO 'Software-Restriction' for HR OU: blocked execution of .exe files from %TEMP% and Downloads
  8. Ran gpupdate /force on test workstations in each OU
  9. Used gpresult /h C:\report.html to generate the Resultant Set of Policy report
  10. Identified and resolved a GPO conflict where a child OU policy was being overridden by the domain policy
  11. Used 'Enforced' setting on the Security-Baseline GPO to ensure it couldn't be blocked by OU administrators
  12. Verified all settings applied correctly across IT, HR, and Sales workstations

Key Findings

  • GPO processing order (LSDOU): Local > Site > Domain > OU — later-processed GPOs win by default
  • Item-Level Targeting in Group Policy Preferences allows granular conditions (group membership, OS, IP range)
  • The 'Enforced' flag prevents child OUs from blocking a GPO and makes it take highest precedence
  • Loopback Processing Mode applies user settings based on computer location, not user OU
  • gpresult /h generates an HTML report showing every applied and denied policy — essential for troubleshooting
  • Security Filtering on a GPO controls which users/computers it applies to within the linked OU
  • GPOs replicate between domain controllers via SYSVOL — replication issues can cause inconsistent policy application
  • Always test GPOs in a pilot OU with a small set of test accounts before domain-wide deployment

Outcome & Lessons Learned

All Group Policy objectives were successfully implemented. Password policies, USB restrictions, desktop standardization, and drive mappings were applied correctly to the appropriate OUs. The most important lesson from this case study was that thorough planning of GPO scope and testing in a staging OU before production deployment prevents widespread issues. A single misconfigured GPO linked at the domain level can affect all users and computers simultaneously — always verify with gpresult before considering a deployment complete.
