Case Study 02 — User Management

Add Domain Users & Organizational Units

CategoryActive Directory
DifficultyBeginner–Intermediate
OS UsedWindows Server 2008 R2

Scenario

After the domain controller was established, the organization needed a structured approach to managing its 40 staff members across three departments: IT, HR, and Sales. The task involved designing an OU hierarchy and creating domain user accounts for each employee.

Overview

Organizational Units (OUs) serve as containers within Active Directory that group users, computers, and groups logically — typically mirroring the organizational structure of the company. This case study covers creating a logical OU tree, creating user accounts with correct attributes, assigning users to security groups for resource access, and applying department-specific Group Policy Objects to each OU.

Case Study Video

Implementation Steps

  1. Opened Active Directory Users and Computers (ADUC) on the Domain Controller
  2. Right-clicked the domain root and selected New > Organizational Unit
  3. Created top-level OUs: IT, HR, Sales, and Computers
  4. Created sub-OUs inside each department OU: Users and Groups
  5. Right-clicked the Users sub-OU for each department and selected New > User
  6. Filled in First Name, Last Name, User Logon Name following naming convention (firstname.lastname)
  7. Set initial password and checked 'User must change password at next logon'
  8. Created Security Groups per department (e.g., GRP-IT, GRP-HR, GRP-Sales)
  9. Added each user to their department security group via the group's Members tab
  10. Verified users could log into workstations using their new domain credentials

Key Findings

  • OUs allow delegation of administrative control — e.g., an HR manager can reset HR user passwords without domain admin rights
  • Naming conventions for user accounts should be documented and enforced consistently from day one
  • Security Groups should be used for resource access rather than assigning permissions to individual users
  • The 'User must change password at next logon' option forces users to set their own private password immediately
  • Disabling unused accounts rather than deleting them preserves audit history and SID-based permissions
  • Account Expiration dates can be set for contractors or temporary staff
  • Moving a user object between OUs changes which GPOs apply to them — test before bulk moves
  • ADUC has a 'Find' feature (Ctrl+F) that is essential for locating objects in large directories

Outcome & Lessons Learned

All 40 user accounts were created and organized within a logical OU structure that mirrors the company's departments. Department-specific GPOs were later linked to each OU to enforce relevant desktop and security settings. The primary lesson was that planning the OU structure on paper before implementing in ADUC saves significant restructuring effort — changing the OU hierarchy after GPOs are linked and permissions are delegated is disruptive and time-consuming.
